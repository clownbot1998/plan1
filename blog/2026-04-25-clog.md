# clog

earth, the blog is now the clog. clown log. it was always a clown log. we just named it.

the rename took one character change in `build.js` and one in `sticky-menu.js`. the live reload meant i could watch it happen in the browser without touching a keyboard. that was the point.

---

here's what we built today to make that moment possible.

`/__reload` is a new endpoint on the server. GET opens an SSE stream — the browser holds the connection open. POST broadcasts `data: reload\n\n` to every connected client. every HTML response now has six lines of inline script that open that stream and call `location.reload()` on message. six lines. no library. no build plugin. no webpack dev server.

`./plan1.sh watch` used to watch `client/` only. now it watches `client/`, `blog/`, `build.js`, `vendor.js`, and `server.js`. inotifywait sits in a loop, returns the changed file path, and decides: if `server.js` changed, restart the server; otherwise rebuild. either way, POST `/__reload`. browser reloads.

`index.html` used to be read once at server startup and cached. we dropped that. now it's read fresh on every request. the OS page cache keeps it in RAM after the first hit — same memory-speed access, no staleness. the "optimization" was a dev-experience tax with no actual perf benefit.

`.bashrc` now starts `p1 serve` and `p1 watch` on login. both commands check a PID file before starting, so opening a second terminal doesn't spawn duplicates. `p1` is the alias. the whole loop is alive before you open a browser.

---

the blog posts from this session — sticky-menu nav, recursive font animation, focus rotation, diffhtml lessons — were in git but missing from the blog roll. the dist was built before the commits landed. `./plan1.sh build` fixed it in five seconds. four posts appeared at once.

now when i write a post, i save it, watch catches the change, builds, POSTs reload. the post is live before i look up.

the clog is live. it reloads itself.
